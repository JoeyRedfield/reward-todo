# Reward Todo 认证重构与注册流设计

日期：2026-06-22

## 背景

`reward-todo` 最近合入了登录改造，后端已经从旧的 Nginx Basic Auth 过渡到应用内 Session 登录，并新增了 Agent / MCP 相关能力。当前还存在两个明显问题：

1. 视觉层面，登录页和账户页的字体、字号层级、版式密度不适合作为稳定产品界面，标题过重，内容结构失衡。
2. 认证能力层面，当前只有基于 `AUTH_INITIAL_USERNAME` / `AUTH_INITIAL_PASSWORD` 的启动初始账号，没有类似 `ezbookkeeping` 的显式注册入口和成体系的注册流程。

本次设计采用“实用版 `ezbookkeeping` 对齐”方案：保留当前项目的轻量业务边界，但将登录/注册的入口组织、视觉基调和注册成功后的工作台初始化方式向 `ezbookkeeping` 靠拢。

## 目标

1. 提供清晰的登录 / 注册双入口，未注册用户可在前端完成注册并直接进入产品。
2. 保留初始账号 bootstrap 机制，确保 `docker compose up` 后无需额外操作即可登录。
3. 将认证页和账户页的视觉语言改为更克制、稳定、可长期维护的产品界面。
4. 让注册第二步具备实际业务价值：不是空表单，而是初始化 reward-todo 的默认项目与模板。

## 非目标

1. 本次不实现邮箱验证、忘记密码、短信验证码、OAuth、2FA。
2. 本次不照搬 `ezbookkeeping` 的货币、语言、周起始日、预置分类等记账领域字段。
3. 本次不改变现有任务奖励主业务 API 的语义。

## 方案总览

### 认证模式

系统同时支持两种入口：

1. 初始账号模式：应用启动时继续根据环境变量确保存在一个初始用户。
2. 用户注册模式：当 `AUTH_ENABLE_REGISTRATION=true` 时，前端展示注册入口，后端开放注册 API。

这两种模式并存，不互斥。初始账号是部署兜底，注册模式是日常使用入口。

### 视觉方向

整体风格从“海报式展示界面”调整为“稳定产品界面”：

1. 中文主字体改为系统无衬线产品字体栈：`"PingFang SC", "Noto Sans SC", "Microsoft YaHei", sans-serif`。
2. 英文、数字和辅助标签使用 `IBM Plex Sans` 或同类中性字体，但不再让压缩展示字体占据大标题。
3. 标题层级收敛，减少超大字号；提升正文可读性和输入区密度一致性。
4. 继续保留少量品牌感颜色，但控制在按钮、链接、当前步骤、焦点态上，不再让整页视觉元素互相竞争。
5. 登录页、注册页、账户页共享一套表单卡片、标题、说明文字、错误提示、步骤条的样式基线。

## 信息架构与页面设计

### 登录页 `/login`

参考 `ezbookkeeping` 的左右结构，但内容收敛为 reward-todo 自身语义：

1. 左侧为品牌说明区。
   - 产品名称
   - 一句清晰说明：这是个人任务/奖励工作台
   - 2 到 3 条极短能力点，如“计划任务”“完成记奖”“支出回收”
2. 右侧为登录卡片。
   - 用户名
   - 密码
   - 登录按钮
   - 会话失效提示
   - 登录失败提示
   - “没有账号？创建账号”入口
3. 移动端改为单列堆叠，品牌说明区缩成顶部说明块。

### 注册页 `/signup`

采用 3 步结构，路径上接近 `ezbookkeeping`，但字段只保留 reward-todo 真正需要的数据。

#### 第 1 步：基础信息

字段：

1. 用户名
2. 显示名称
3. 邮箱
4. 密码
5. 确认密码

规则：

1. 用户名必填，唯一，使用现有用户名规范化逻辑。
2. 显示名称必填，用于界面展示；若将来不再单独展示，也保留字段以避免再次迁移。
3. 邮箱必填，格式必须合法，且全局唯一。
4. 密码与确认密码必须一致，并满足当前系统的最小强度要求。

页面同时提供“已有账号？去登录”入口。

#### 第 2 步：初始化工作台

这一页对应 `ezbookkeeping` 的“预设分类”思想，但落到 reward-todo 的业务对象上。

字段与交互：

1. 开关：是否创建默认工作台预设
2. 若开启，显示将创建的默认项目与模板摘要

默认预设建议：

1. 项目：学习
   - 模板：背单词 20 分钟 / 奖励 8 元
   - 模板：深度阅读 30 分钟 / 奖励 12 元
2. 项目：运动
   - 模板：力量训练 30 分钟 / 奖励 15 元
   - 模板：拉伸 15 分钟 / 奖励 6 元
3. 项目：生活
   - 模板：整理房间 20 分钟 / 奖励 10 元

这些预设的目的不是“产品推荐”，而是让新用户第一次进入 `/today` 与 `/projects` 时不是空白。

#### 第 3 步：完成

成功态说明：

1. 账号创建成功
2. 是否已初始化默认项目
3. 自动登录结果
4. “进入今日面板”主按钮

默认行为是注册成功后先建立会话，再进入第 3 步完成页；用户点击“进入今日面板”后进入 `/today`。

### 账户页 `/account`

账户页继续保留现有功能，但重做视觉结构：

1. 顶部改为普通页标题与简短描述，不再使用超大展示标题。
2. “账户资料”“最近登录”“密码修改”“令牌管理”按常规产品后台布局排布。
3. 表单宽度、行高、按钮位置统一，减少空白浪费。
4. 字段标签与数值字号分离，弱化装饰性大数字。

## 后端设计

### 用户模型

`User` 新增字段：

1. `display_name`
2. `email`

约束：

1. `display_name` 非空
2. `email` 唯一
3. 历史初始账号在迁移时自动补齐：
   - `display_name` 默认为用户名
   - `email` 使用占位规则，例如 `<username>@local.invalid`

占位邮箱只用于兼容迁移，不用于对外展示文案。

### 新增配置

新增：

1. `AUTH_ENABLE_REGISTRATION`

兼容保留：

1. `AUTH_INITIAL_USERNAME`
2. `AUTH_INITIAL_PASSWORD`
3. 旧别名 `APP_BASIC_AUTH_USER`
4. 旧别名 `APP_BASIC_AUTH_PASSWORD`

行为：

1. `AUTH_ENABLE_REGISTRATION` 默认值为 `true`。
2. 若关闭注册，前端隐藏注册入口，后端注册接口返回 403。

### 注册 API

新增 `POST /api/auth/register`

请求体：

1. `username`
2. `display_name`
3. `email`
4. `password`
5. `confirm_password`
6. `create_default_workspace`

响应：

1. 成功时返回当前用户信息
2. 同时设置 Session Cookie

错误场景：

1. 用户名已存在
2. 邮箱已存在
3. 两次密码不一致
4. 密码不符合要求
5. 注册功能关闭

### 默认工作台初始化

在注册服务层中新增“创建默认项目与模板”能力。该能力：

1. 仅在注册成功且 `create_default_workspace=true` 时执行
2. 与用户创建放在同一事务中
3. 任一步失败则整体回滚，避免出现“账号建好了，但预设只创建一半”的状态

## 前端设计

### 路由与状态

新增路由：

1. `/signup`

认证上下文新增能力：

1. `register(payload)`，成功后写入 `user`

API 客户端新增：

1. `register`

### 可复用组件

建议抽出认证页共用壳层，例如：

1. `AuthShell`
2. `AuthAside`
3. `AuthCard`
4. `AuthSteps`

目标不是为抽象而抽象，而是确保登录页与注册页在布局和样式上天然一致。

### 文案原则

1. 使用产品语言，不用海报式修辞。
2. 按钮文案与结果文案保持一致，例如“创建账号”对应“账号创建成功”。
3. 错误提示直接说明问题，不做情绪化表达。

## 环境变量与启动说明

README 与示例环境文件需要明确两套路径：

### 路径 1：直接用初始账号登录

适合本地快速启动：

1. `.env` 中设置 `AUTH_INITIAL_USERNAME`
2. `.env` 中设置 `AUTH_INITIAL_PASSWORD`
3. `docker compose up` 后直接使用该账号登录

### 路径 2：开放注册

适合多用户体验验证：

1. `.env` 中设置 `AUTH_ENABLE_REGISTRATION=true`
2. 应用启动后可从登录页进入注册页
3. 注册完成后自动进入系统

同时需要明确：旧字段 `APP_BASIC_AUTH_USER` / `APP_BASIC_AUTH_PASSWORD` 仍兼容读取，但文档主路径统一迁移到 `AUTH_INITIAL_*`。

## 测试与验证

### 后端测试

新增或更新测试覆盖：

1. 注册成功并自动登录
2. 重复用户名失败
3. 重复邮箱失败
4. 注册关闭时失败
5. 选择初始化默认工作台时成功创建项目与模板
6. 初始账号兼容旧环境变量仍可启动

### 前端测试

新增或更新测试覆盖：

1. 登录页存在注册入口
2. 注册页步骤流转正常
3. 基础信息校验正常
4. 提交成功后进入完成态或直接跳转 `/today`
5. 注册关闭时不显示注册入口

### 集成验证

完成实现后执行：

1. 本地测试套件
2. `docker compose up --build`
3. 浏览器手工验证登录、注册、自动登录、账户页展示

## 实施边界

本次实现预计会修改：

1. 后端认证 schema / service / router / model / migration / config
2. 前端 auth context / api client / 路由 / 登录页 / 新注册页 / 账户页样式
3. `README.md`
4. `.env.example`

不会扩展到：

1. 邮箱验证
2. 忘记密码
3. 第三方登录
4. 更复杂的用户资料中心

## 验收标准

满足以下条件即认为本次目标完成：

1. 旧 `.env` 与新 `.env` 写法都能让 `docker compose up` 正常启动。
2. 使用初始账号可以登录。
3. 登录页可以进入注册页。
4. 新用户可以注册、自动登录，并在进入系统后看到非空默认工作台。
5. 登录页与账户页的字体、间距、标题层级较当前实现明显收敛，整体更接近稳定产品界面而非展示海报。
