# Reward Todo 直接添加独立任务设计

日期：2026-06-24

## 背景

当前系统里，日任务只能通过“项目模板 -> 加入今日/当前日期”的路径创建。这个模型适合长期重复任务，但不适合临时任务、一次性任务或用户尚未整理成模板的任务。

这带来两个直接问题：

1. 用户想记一条今天要做的任务时，必须先去 `/projects` 创建项目和模板，再回到 `/today` 加入当天，路径过长。
2. `/today` 页面虽然已经支持选中任意日期作为工作台，但“添加任务”仍然强绑定模板来源，无法直接把临时任务落到当前选中日期。

本次设计目标是在不破坏现有模板任务链路的前提下，为 `/today` 增加一条“直接添加独立任务”的路径，并补齐首版必须的删除与账本一致性规则。

## 目标

1. 用户可以在 `/today` 当前选中日期下直接创建一条独立任务，而不必先创建项目或模板。
2. 独立任务与模板任务共享同一套日任务列表、完成/撤销完成、奖励统计与奖励账本。
3. 独立任务首版支持删除，且删除已完成任务时会自动冲回仍然有效的奖励金额。
4. 现有模板任务创建能力、Today 页面模板快速加入能力、对外 API 与 MCP 能力保持兼容，并在必要处做最小扩展。

## 非目标

1. 本次不实现“把独立任务保存成模板”。
2. 本次不实现独立任务编辑。
3. 本次不为模板任务增加删除能力。
4. 本次不调整 `/projects` 页的项目与模板管理模型。
5. 本次不新增独立任务归档、软删除或回收站能力。

## 方案总览

采用“同一张 `daily_tasks` 表承载两类任务”的方案：

1. 模板任务继续沿用现有数据流，由模板派生，保留模板来源。
2. 独立任务直接在 `/today` 创建，不要求项目与模板来源。
3. 用 `task_template_id = null` 表示独立任务，不新增 `source_type` 等来源枚举字段。
4. 首版独立任务的 `project_id` 也允许为 `null`，避免为了独立任务强行引入默认项目。
5. 复用现有 `create_daily_task` 主语义接口，通过请求体模式区分“模板任务创建”和“独立任务创建”。

这样可以保持“创建日任务”这个业务动作只有一个主入口，同时把“来源是模板还是手动录入”收敛在后端校验与数据字段上。

## 数据语义

### 日任务来源

日任务分成两类：

1. 模板任务
   - `task_template_id` 有值
   - `project_id` 继承模板所属项目
   - `name_snapshot`、`estimated_duration_minutes_snapshot`、`reward_amount_snapshot` 继续按当前逻辑写入快照
2. 独立任务
   - `task_template_id = null`
   - `project_id = null`
   - `name_snapshot`、`estimated_duration_minutes_snapshot`、`reward_amount_snapshot` 直接来自 `/today` 表单输入

约束：

1. 首版以 `task_template_id = null` 作为独立任务唯一判定条件。
2. 返回给前端、API token 与 MCP 客户端的日任务数据中，`project_id` 和 `task_template_id` 都需要允许为 `null`。
3. 独立任务允许同一日期内重名重复创建，不增加名称去重规则。

### 奖励金额与时长

独立任务表单规则：

1. 名称必填。
2. 预计时长必填，且默认留空，不预填系统默认值。
3. 奖励金额选填；前端展示占位 `0.00`，并提示“留空按 ¥0.00 处理”。
4. 奖励金额留空时，前端直接提交 `0` 分，而不是把空值交给后端兜底。

## 页面交互设计

### `/today` 页面结构

`/today` 的“快速加任务”面板调整为两段：

1. 面板顶部常驻“直接添加任务”表单
2. 表单下方保留现有“从启用模板快速加入当前选中日期”的列表

行为：

1. 直接添加任务表单默认展开，不做折叠态。
2. 即使当前没有任何项目或模板，直接添加任务表单仍然显示。
3. 当模板列表为空时，仅模板区域显示“当前没有可直接加入台账的启用模板”，不影响直接添加任务表单。

### 直接添加任务表单

字段：

1. 任务名称：文本输入，必填
2. 预计时长（分钟）：数字输入，必填
3. 奖励金额（元）：数字输入，选填，占位 `0.00`

提交规则：

1. 任务会加入“当前选中日期”，而不是固定加入今天。
2. 提交成功后，名称、时长、奖励三个字段全部清空。
3. 提交成功后，当前选中日期保持不变。
4. 提交成功后，刷新当前日期的任务列表、日期摘要与奖励摘要。

### 日任务列表呈现

独立任务在“当日任务”列表中需要有明确标识，例如标签 `独立任务`。

按钮规则：

1. 模板任务
   - 未完成：维持现有 `完成`
   - 已完成：维持现有 `撤销完成`
   - 不显示删除
2. 独立任务
   - 未完成：显示 `完成 + 删除`
   - 已完成：显示 `撤销完成 + 删除`

排序规则：

1. 首版不调整列表排序。
2. 独立任务继续按现有 `created_at asc, id asc` 顺序展示，因此新建任务通常出现在列表末尾。

## 接口设计

### 创建日任务

继续复用 `POST /api/daily-tasks`，但扩展为两种创建模式。

模板任务模式：

```json
{
  "date": "2026-06-24",
  "task_template_id": 12,
  "estimated_duration_minutes": 30,
  "reward_amount": 1200
}
```

独立任务模式：

```json
{
  "date": "2026-06-24",
  "name": "临时复盘",
  "estimated_duration_minutes": 20,
  "reward_amount": 0
}
```

后端严格做二选一校验：

1. 模板任务必须提供 `task_template_id`，且不能再传独立任务用的 `name`
2. 独立任务必须提供 `name`，且 `task_template_id` 必须为空或不传
3. 两边都传，或两边都不传，一律返回 `400`

原因：

1. 维持“创建日任务”作为统一业务动作的语义。
2. 避免为来源差异新增两个高度相似的创建接口。
3. 保持前端、API token、MCP、skill 脚本的能力模型尽量一致。

### 删除日任务

新增：

1. `DELETE /api/daily-tasks/{task_id}`

删除规则：

1. 仅允许删除独立任务。
2. 如果目标任务 `task_template_id` 不为空，则返回业务错误，例如“只有独立任务支持删除”。
3. 成功删除返回 `204 No Content`。

删除确认文案：

1. 未完成独立任务：`确认删除任务「xxx」吗？`
2. 已完成独立任务：`确认删除已完成任务「xxx」吗？删除后会扣回已发放奖励。`

### MCP 与对外能力

对外能力同步扩展：

1. HTTP API 新增 `DELETE /api/daily-tasks/{task_id}`
2. MCP tools 新增 `delete_daily_task`
3. 现有 MCP `create_daily_task` schema 更新为支持两种创建模式
4. 相关 skill / 脚本文档需要同步反映独立任务创建和删除能力

## 服务层与账本规则

### 完成与撤销完成

独立任务沿用现有日任务状态与奖励规则：

1. 完成任务时，如果该任务当前关联流水净额小于等于 `0`，写入一条 `earn`
2. `reason = name_snapshot`
3. 撤销完成时继续写入负向 `adjust`
4. `reason = reopen:${name_snapshot}`

这样独立任务与模板任务在“完成/撤销完成/日奖励统计”上的行为保持一致。

### 删除独立任务时的奖励处理

删除未完成独立任务：

1. 通常没有奖励流水，直接删除任务即可。

删除已完成独立任务：

1. 允许删除，即使这会让当前奖励余额变成负数。
2. 先计算该任务当前关联奖励流水的净额。
3. 如果净额大于 `0`，补一条负向 `adjust` 流水：
   - `amount = -净额`
   - `reason = delete:${name_snapshot}`
   - `daily_task_id = null`
4. 将这条任务已有历史流水的 `daily_task_id` 全部置为 `null`
5. 最后硬删除该独立任务

采用这套规则的原因：

1. 保留真实奖励流水历史，不删除过去的 `earn` / `adjust` 记录。
2. 避免硬删除任务后留下悬空外键引用。
3. 只按“当前净收益”冲回，避免任务经历过多次完成/撤销后账本被重复扣减。

## 数据库与 Schema 调整

首版需要允许独立任务脱离模板和项目存在，因此需要调整：

1. `daily_tasks.project_id` 改为可空
2. `daily_tasks.task_template_id` 改为可空
3. 当前基于 `(date, task_template_id)` 的唯一约束需要重新评估

约束建议：

1. 模板任务仍需保留“同一模板同一天只能加入一次”的规则
2. 独立任务不参与该去重约束

实现上更适合迁移为“仅对非空 `task_template_id` 生效”的数据库约束或等价机制，而不是继续使用当前对 `task_template_id` 强制非空的模型假设。

## 错误处理

1. 创建独立任务时，名称为空、时长非法、奖励金额格式非法，继续返回明确校验错误。
2. 删除模板任务时，后端明确返回业务错误，不依赖前端隐藏按钮作为唯一保护。
3. 删除已完成独立任务时，即使余额不足或会变成负数，也不拦截删除。
4. 创建、删除成功后如果面板刷新失败，前端仍应展示刷新失败提示，避免静默错误。

## 测试与验证

### 后端测试

需要补充以下场景：

1. 独立任务可以通过 `name + date + estimated_duration_minutes + reward_amount` 创建成功。
2. 模板任务现有创建链路继续可用。
3. 创建请求混传模板字段与独立任务字段会被拒绝。
4. 独立任务完成后会写入 `earn`，`reason` 为任务名称。
5. 独立任务删除未完成任务时直接消失。
6. 独立任务删除已完成任务时：
   - 会按净额补一条 `delete:` 负向流水
   - 原有关联流水 `daily_task_id` 被解绑
   - 任务记录被删除
7. 模板任务删除请求会被拒绝。
8. 日任务读取、奖励汇总、奖励流水在 `project_id` / `task_template_id` 可空后仍可正常工作。

### 前端测试

需要补充以下场景：

1. `/today` 页面显示直接添加任务表单。
2. 没有任何模板时，直接添加任务表单仍显示。
3. 提交独立任务时，奖励留空会以 `0` 提交。
4. 提交成功后表单清空，并且停留在当前选中日期。
5. 独立任务在列表中带有 `独立任务` 标识。
6. 未完成独立任务显示 `完成 + 删除`。
7. 已完成独立任务显示 `撤销完成 + 删除`。
8. 删除独立任务时会触发确认流程，并在成功后刷新面板。
9. 模板任务仍然没有删除按钮。

### MCP / API 测试

需要补充以下场景：

1. MCP `create_daily_task` 支持独立任务创建参数。
2. MCP 新增 `delete_daily_task`。
3. 对模板任务调用删除时返回明确业务错误。

### 手工验证

至少验证以下路径：

1. 在 `/today` 选中某个非今天日期，直接添加独立任务，任务加入该日期。
2. 奖励金额留空时，任务以 `¥0.00` 展示并可正常完成。
3. 删除未完成独立任务后，列表与摘要立即刷新。
4. 完成独立任务后再删除，账本新增 `delete:` 扣回流水，余额正确变化。
5. 模板快速加入仍可用，模板任务仍不可删除。

## 取舍与理由

本次没有引入新的任务来源字段，也没有为独立任务建立单独表，原因是：

1. 现有 `daily_tasks` 已经具备表达独立任务所需的快照字段。
2. 用 `task_template_id = null` 判定独立任务，语义直接，且变更面小于新增来源枚举。
3. 首版只支持删除独立任务，可以先在最小范围验证这条新数据链路，而不一次性重写整个日任务管理模型。

本次也没有引入独立任务软删除，原因是：

1. `daily_tasks` 当前没有删除状态字段。
2. 首版需求是“录错时可以撤回”，而不是日任务回收站。
3. 在已经保留账本历史流水的前提下，任务硬删除足以满足产品语义。

## 实施边界

本次实现仅覆盖：

1. `/today` 页面直接添加独立任务
2. 独立任务的数据模型、创建、完成、撤销完成与删除
3. 删除能力在 HTTP API、MCP 与相关客户端 schema 中的同步扩展
4. 对应前后端与 MCP 测试

不包含：

1. 独立任务编辑
2. 独立任务保存成模板
3. 模板任务删除
4. 独立任务归档或回收站
