# Reward Todo 项目与模板删除恢复及默认值兜底设计

日期：2026-06-22

## 背景

当前 `/projects` 页面已经支持创建项目、创建模板，以及把模板加入今日任务，但还存在两个直接影响使用的问题：

1. 项目和模板缺少“删除后可恢复”的长期管理能力，用户只能继续保留不再使用的条目，工作区会逐渐变脏。
2. 模板创建表单里的“默认时长”和“默认奖励”在不填写时没有回退到系统默认值，导致用户得到 `0` 分钟、`0` 元或直接被校验拦住，不符合“留空使用默认配置”的预期。

本次设计目标是在不引入额外复杂数据模型的前提下，为项目和模板补齐可恢复删除能力，并统一模板默认值行为。

## 目标

1. 项目支持“删除”和“恢复”，删除后的项目不会出现在默认工作区主列表中，但可以持续恢复。
2. 模板支持“删除”和“恢复”，删除后的模板不会继续用于今日任务创建，但可以持续恢复。
3. 模板创建时，默认时长和默认奖励留空会自动使用系统默认值，而不是提交 `0` 或报出与预期不符的结果。
4. 保持现有后端模型和接口结构尽量稳定，优先复用已有字段和更新接口。

## 非目标

1. 本次不实现真正物理删除，不从数据库永久移除项目和模板。
2. 本次不新增完整回收站、操作日志、删除审计或批量恢复功能。
3. 本次不调整今日任务、奖励账本或账户页的交互。
4. 本次不引入模板编辑表单，仅处理已有的创建、删除、恢复与加入今日能力。

## 方案总览

采用“软删除即持久可恢复”方案，复用现有字段表达删除状态：

1. 项目删除使用 `task_projects.status = "archived"`。
2. 项目恢复使用 `task_projects.status = "active"`。
3. 模板删除使用 `task_templates.is_active = false`。
4. 模板恢复使用 `task_templates.is_active = true`。

前端文案仍然面向用户使用“删除”和“恢复”，不暴露“归档”这种内部实现术语。这样既满足“删除后可撤销”的长期语义，也避免新增 `deleted_at` 等新字段和迁移成本。

## 数据语义

### 项目

项目有两个本次需要关注的状态：

1. `active`：正常可用，显示在可用项目列表中，可被选中用于创建模板。
2. `archived`：视为已删除，默认不参与主工作流，但保留在“已删除项目”分组中，可恢复。

约束：

1. 已删除项目不能作为当前可选项目参与模板创建。
2. 已删除项目下的模板数据保留，不额外迁移、不额外复制。
3. 项目恢复后，其下模板按原有状态继续显示和使用。

### 模板

模板沿用现有 `is_active` 字段：

1. `true`：正常可用，显示在可用模板列表中，可加入今日任务。
2. `false`：视为已删除，显示在“已删除模板”分组中，可恢复。

约束：

1. 已删除模板不提供“加入今日”入口。
2. 已删除模板恢复后，重新出现在可用模板列表，并继续可用于创建今日任务。

### 默认值

模板创建表单引入两个系统默认值：

1. 默认时长：`20` 分钟
2. 默认奖励：`1200` 分

规则：

1. 如果输入框为空字符串，则提交时回退到对应默认值。
2. 如果用户明确输入非法值，则仍然走校验报错，而不是静默纠正。
3. 如果用户明确输入 `0`：
   - 时长 `0` 仍然非法，因为模板时长必须大于 `0`
   - 奖励 `0` 合法，表示该模板不奖励积分

## 页面交互设计

### 项目区域

项目面板拆分为两个分组：

1. 可用项目
2. 已删除项目

可用项目条目提供：

1. 选择
2. 删除

已删除项目条目提供：

1. 恢复

行为：

1. 默认自动选中第一个可用项目。
2. 如果当前选中的项目被删除，则自动切换到下一个可用项目。
3. 如果没有任何可用项目，则模板创建表单禁用，模板列表清空。
4. 恢复项目后只刷新列表，不强制抢占当前选中项。

删除确认：

1. 删除操作先要求一次明确确认，避免误点。
2. 删除成功后展示成功提示，例如“项目已删除，可在已删除项目中恢复”。
3. 恢复成功后展示提示，例如“项目已恢复”。

### 模板区域

模板列表同样拆分为两个分组：

1. 可用模板
2. 已删除模板

可用模板卡片提供：

1. 加入今日
2. 删除

已删除模板卡片提供：

1. 恢复

行为：

1. 已删除模板不展示“加入今日”。
2. 删除成功后展示提示，例如“模板已删除，可在已删除模板中恢复”。
3. 恢复成功后展示提示，例如“模板已恢复”。

### 模板创建表单

默认时长和默认奖励字段继续展示为可编辑数字输入框，但语义改为“可留空”：

1. 输入框允许为空。
2. 提交时若为空，前端自动补为 `20` 和 `1200`。
3. 表单附近增加轻提示，说明“留空时使用默认：20 分钟 / 1200 分”。
4. 用户输入非法值时，仍显示明确错误文案。

## 接口与状态管理设计

### 后端接口

不新增删除专用路由，直接复用已有更新接口：

1. `PATCH /api/task-projects/{project_id}`
   - 删除：`{ "status": "archived" }`
   - 恢复：`{ "status": "active" }`
2. `PATCH /api/task-templates/{template_id}`
   - 删除：`{ "is_active": false }`
   - 恢复：`{ "is_active": true }`

原因：

1. 现有后端已经支持项目状态更新和模板激活状态更新。
2. 新增删除/恢复专用接口会重复语义而不会显著提升清晰度。

### 前端 API Client

现有 `updateProject` 直接复用。

新增：

1. `updateTaskTemplate(templateId, payload)`

### `useProjectsBoard` 状态管理

新增动作：

1. `archiveProject`
2. `restoreProject`
3. `archiveTemplate`
4. `restoreTemplate`

状态流：

1. 加载项目页时拉取所有项目。
2. 前端按 `status` 将项目分为可用和已删除两组。
3. 选中项目后拉取该项目下所有模板。
4. 前端按 `is_active` 将模板分为可用和已删除两组。
5. 删除或恢复任一项目/模板后刷新相关列表，并重算当前选中项目。

### 默认值兜底落点

默认值兜底放在前端提交层实现，而不是修改后端 schema 允许空值：

1. 前端提交前将空字符串转换成默认值。
2. 后端继续保持现有数值校验，防止绕过前端的非法请求进入系统。

这样能把“界面留空表示使用默认”的产品语义限制在 UI 层，不污染 API 的数值合同。

## 错误处理

1. 当项目删除或恢复失败时，保留当前界面状态并展示错误提示。
2. 当模板删除或恢复失败时，保留当前界面状态并展示错误提示。
3. 当恢复或删除后重新拉取列表失败时，优先保留成功动作结果提示，同时展示刷新失败提示。
4. 如果当前选中项目在刷新后已不存在可用状态，则自动切到首个可用项目；如果没有可用项目，则清空模板区域。

## 测试与验证

### 前端测试

需要补充以下场景：

1. 模板创建时默认时长和默认奖励留空，会提交 `20` 和 `1200`。
2. 点击项目删除会调用 `updateProject(projectId, { status: "archived" })`。
3. 点击项目恢复会调用 `updateProject(projectId, { status: "active" })`。
4. 点击模板删除会调用 `updateTaskTemplate(templateId, { is_active: false })`。
5. 点击模板恢复会调用 `updateTaskTemplate(templateId, { is_active: true })`。
6. 删除当前选中的项目后，会自动切换到下一个可用项目。
7. 已删除模板不展示“加入今日”按钮。

### 后端测试

需要补充以下场景：

1. 项目状态可从 `active` 更新到 `archived`，再恢复到 `active`。
2. 模板 `is_active` 可从 `true` 更新到 `false`，再恢复到 `true`。
3. 模板停用后不能创建今日任务，恢复后可以再次创建今日任务。

### 手工验证

至少验证以下路径：

1. 创建模板时清空默认时长和默认奖励，成功生成 `20` 分钟、`1200` 分的模板。
2. 删除项目后，该项目移入“已删除项目”，并且模板创建表单切换到下一个可用项目。
3. 恢复项目后，项目重新回到可用列表。
4. 删除模板后，该模板移入“已删除模板”并不可加入今日。
5. 恢复模板后，该模板重新可加入今日。

## 取舍与理由

本次没有引入单独的 `deleted_at` / `deleted_by` 字段，也没有增加物理删除能力，原因是：

1. 当前产品规模较小，现有状态字段已经足以表达“从主工作流移除但可恢复”。
2. 用户的核心诉求是“删除后可撤销”，而不是审计级回收站。
3. 复用现有模型可以减少迁移、接口和查询层的复杂度，更适合这次迭代。

## 实施边界

本次实现仅覆盖：

1. `/projects` 页的项目与模板删除/恢复交互
2. 创建模板时的默认值兜底
3. 对应前后端测试

不包含：

1. 真删除接口
2. 模板编辑功能
3. 回收站独立页面
